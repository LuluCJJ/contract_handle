# 网银权限预审：Prompt vNext Demo 实施方案

**适用阶段**：下周领导对标前的 demo 阶段  
**模型约束**：内部部署 Qwen 32B / 72B  
**核心目标**：在“通用性”和“精准性”之间做现实权衡，给当前 Demo 一套更强但不过重的 Prompt 方案。

---

## 1. 先回答最关键的担心

你的担心是对的：

> 如果我们完全按前面的完整框架推进，系统会比现在复杂太多。

对于 demo 阶段，这种复杂度可能带来三个问题：

1. 模型输出不稳定
2. 代码改造范围过大
3. 领导看不到“多出来的复杂度”到底值不值

所以当前阶段最合适的策略不是“全量重构 Prompt 体系”，而是：

> 保留现有三段式主流程，只把最关键的业务逻辑补进去。

也就是仍然保留：

- `global_document_extraction`
- `semantic_risk_analyzer`
- `multi_doc_summary`

但把它们升级成“更有业务感知”的 vNext 版本。

---

## 2. Demo 阶段的取舍原则

### 2.1 优先精准，不优先绝对通用

对下周演示来说，最重要的是让领导看到：

- 系统能理解开通/注销这类关键场景
- 系统能抓住权限、介质、账户这些真正的风险点
- 系统不会把明显的灰区假装判清楚

所以当前阶段应优先做：

- 场景识别
- 权限范围四分类归一
- 介质条件矩阵判断
- 账号/账户状态检查
- 人工确认提示

而不是优先做：

- 所有字段全覆盖
- 所有银行全泛化
- 所有场景全自动无歧义推理

### 2.2 优先增强现有 Prompt，不优先拆成 5 个 Prompt

完整未来态可以拆成：

- fact extraction
- normalization
- field audit
- cross doc
- report aggregator

但 demo 阶段不建议这么拆。

推荐策略：

- `global_document_extraction`：抽取 + 初步归纳场景/动作
- `semantic_risk_analyzer`：字段簇级检查
- `multi_doc_summary`：报告汇总

如有必要，再新增一个：

- `cross_document_audit_checker`

## 3. Demo 阶段最值得改的 Prompt

## 3.1 `global_document_extraction` vNext

### 当前主要问题

- 过于泛化
- 没有显式感知开通/注销
- 对介质和动作表达不够
- 没有要求输出证据摘要

### vNext 应增加的核心任务

1. 识别当前业务场景
2. 识别当前动作类型
3. 识别动作作用对象
4. 提取更完整的权限、介质、账户信息
5. 输出简单证据摘要

### 最小新增输出建议

```json
{
  "scenario_type": "OPEN|CANCEL|UNKNOWN",
  "action_type": "OPEN_PERMISSION|CANCEL_PERMISSION|OPEN_MEDIA|CANCEL_MEDIA|UNKNOWN",
  "action_summary": "本次申请为注销某名义用户在 ICBC 平台下的支付/授权权限，并注销对应 Token 介质。",
  "evidence_summary": "文档多处出现‘注销’、‘取消权限’、‘是否注销介质=是’等表述。"
}
```

### 为什么这是最小有效增强

因为它能立刻提升：

- 场景理解能力
- 后续检查器的上下文
- Demo 的可解释性

---

## 3.2 `semantic_risk_analyzer` vNext

### 当前主要问题

- 还停留在泛化语义比较
- 没有显式区分检查模式
- 没有“人工确认”机制
- 没有开通/注销方向意识

### vNext 要补的不是更多长篇描述，而是更明确的检查框架

建议让它围绕 5 类字段簇输出：

- `business_scenario`
- `subject`
- `permission`
- `media`
- `account`

并强制每条检查项输出：

```json
{
  "field_group": "...",
  "check_mode": "...",
  "scenario_type": "...",
  "manual_confirmation_required": true,
  "reason_code": "...",
  "detail": "..."
}
```

### vNext 最该补进来的规则

1. 权限范围四分类对齐
2. 开通场景的超配检查
3. 注销场景的少销/误销检查
4. 当前平台已有介质的条件矩阵判断
5. 介质注销动作一致性
6. 账户状态反向风险检查
7. 名义用户/申请人映射与多文档统一性

### 这里的关键不是“让模型做全部判断”

而是：

- 让模型更清楚地表达自己在检查哪一类问题
- 把需人工确认的灰区显式打出来

---

## 3.3 `multi_doc_summary` vNext

### 当前主要问题

- 更像通用总结器
- 对“需人工确认”承接不够
- 对场景化风险表达不够

### vNext 应重点增强

1. 用一句话总结本次业务场景
2. 汇总最关键的跨文档冲突
3. 单独列出需人工确认点
4. 用更业务化的话术表达风险

### 推荐新增输出

```json
{
  "overall_status": "HIGH_RISK|MED_RISK|LOW_RISK|ZERO_RISK",
  "aggregator_summary": "...",
  "scenario_summary": "本次为权限注销场景，涉及 1 名义用户、1 个 Token 介质、1 个企业账户。",
  "manual_confirmation_items": [
    "系统存在介质，但附件中未明确体现介质编号，建议人工确认。"
  ],
  "risk_insights": [
    "注销对象范围与系统登记权限存在偏差，存在误销风险。"
  ]
}
```

## 4. Demo 阶段是否要新增 `cross_document_audit_checker`

### 我的建议：可加，但不是必须

如果你们下周展示时特别想强调“多文档交叉校验能力”，那新增一个专门的 `cross_document_audit_checker` 是有展示价值的。

但如果时间紧，完全可以先不做成独立 Prompt，而是：

- 在后端先做最简单的跨文档规则检查
- 把结果塞到现有 `cross_validation_checks`

也就是说：

- 产品化阶段：建议独立 Prompt / 独立模块
- Demo 阶段：可以先由规则层承担最关键的交叉校验

## 5. 对 Qwen 32B / 72B 的现实建议

### 5.1 适合它做的

- 单文档结构化抽取
- 权限语义归一
- 场景识别
- 账户名称/主体匹配判断
- 总结与解释

### 5.2 不适合一开始就让它做得太重的

- 超大、多层嵌套 JSON
- 一个 Prompt 同时做抽取、映射、检查、交叉校验、总结
- 特别长、特别细、特别苛刻的多轮逻辑输出

### 5.3 更现实的策略

- 用模型做“识别、归一、解释”
- 用代码做“矩阵、枚举、精确比对、交叉一致性”

这和你们当前内部部署模型的现实约束是更匹配的。

## 6. Demo 前建议做 / 不建议做

## 6.1 建议做

1. 改 `global_document_extraction`
   - 增加场景与动作识别
2. 改 `semantic_risk_analyzer`
   - 增加字段簇、检查模式、人工确认标记
3. 在规则层补最关键矩阵
   - 权限四分类对照
   - 介质有无/注销条件矩阵
   - 账户状态反向风险
4. 改 `multi_doc_summary`
   - 增加场景总结和人工确认项

## 6.2 不建议现在做

1. 全量 Schema 重构
2. 全量 Prompt 体系拆分
3. 所有字段全覆盖
4. 所有场景全支持
5. 大规模前端重做

## 7. 推荐的 Demo 版实施策略

最现实的一版做法是：

### A. 结构上

仍然保留现有主流程：

- 文档解析
- 文档抽取
- 硬比对
- 语义检查
- 汇总报告

### B. 语义上

补进 4 个关键能力：

- 场景识别
- 动作识别
- 字段簇级检查
- 人工确认输出

### C. 展示上

重点展示 3 个亮点：

1. 系统能区分开通和注销
2. 系统能抓权限、介质、账户这些高风险对象
3. 系统能明确告诉人“这里我不建议自动放过，需人工确认”

---

## 8. 一句话总结

对当前 Demo 最合适的 Prompt 路线不是：

> 做成未来最终版的通用智能审计平台

而是：

> 在现有三段式主流程上，最小化补强“场景、动作、关键规则、人工确认”四类能力，让领导能看见这套逻辑已经具备业务落点。
