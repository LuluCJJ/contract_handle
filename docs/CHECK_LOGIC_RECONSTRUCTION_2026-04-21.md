# 网银权限申请预审：检查逻辑重构表

**来源依据**：`网银权限申请信息对照表.xlsx`  
**文档定位**：将业务整理出的字段说明、申请文档映射关系、审核要求，重构为面向系统设计与 Prompt 设计的检查逻辑基线。  
**目标**：明确哪些逻辑属于抽取、哪些属于映射、哪些属于硬规则、哪些属于语义检查、哪些属于多文档交叉校验。

---

## 1. 核心结论

这份 Excel 不是普通字段字典，而是一个混合了三层信息的业务规则源：

1. 字段定义
2. 申请文档映射关系
3. 审核策略与判定逻辑

因此，后续系统设计不能只把它当成“抽取字段列表”，而应当把它视为：

> 面向银行申请材料预审的字段级检查逻辑总表

从系统实施角度看，当前项目需要把这张表拆解为以下 5 类能力：

1. 文档事实抽取
2. 字段映射与语义归一
3. 单文档检查
4. 多文档交叉校验
5. 总体风险汇总与表达

---

## 2. 检查逻辑的统一类型体系

根据 Excel 中出现的审核要求，可以归纳出以下统一的检查类型。

### 2.1 一致性比对

目标：校验申请文档侧信息与系统 E-Flow 基准信息是否一致。

典型适用字段：

- 申请人
- 名义用户
- 账号
- 账户名称
- 账户状态
- 当前平台已有介质
- 权限范围

### 2.2 完整性检查

目标：检查关键字段是否缺失、漏填、空缺、格式异常。

典型适用字段：

- 名义用户
- 账号
- 权限范围
- 介质相关字段
- 多文档申请人信息

### 2.3 反向审视

目标：不仅看“是否一致”，还看是否存在“系统不允许但文档仍在办理”的异常。

典型适用字段：

- 账户状态
  - 系统为失效，但文档按生效状态办理
- 权限范围
  - 文档申请超出现有批准范围
- 介质信息
  - 文档表现出加挂/复用迹象，但系统无对应信息

### 2.4 多文档交叉校验

目标：检查多份申请材料之间是否存在前后不一致、互相冲突、信息矛盾。

典型适用字段：

- 姓名
- 工号
- 部门
- 证件信息
- 账号
- 账户状态
- 空白介质使用情况

### 2.5 合规合理性审视

目标：检查申请内容是否合理、是否超配、是否与业务场景匹配。

典型适用字段：

- 权限范围
- 权限子类
- 介质申请
- 业务场景

### 2.6 人工确认触发

目标：当信息不完整、映射不确定、材料之间存在灰区矛盾时，不给出假确定结论，而是明确标为需人工确认。

典型适用场景：

- 文档有介质信息，系统无值
- 系统有介质信息，文档无值
- 文档字段与系统字段不完全映射
- 文档中提到空白介质，但系统字段为空

## 3. 一级模块重构视图

### 3.1 选择业务场景

涉及字段：

- 需求大类
- 业务场景

重构后的系统含义：

- 这是整个预审链路的“业务上下文锚点”
- 决定后续权限申请、介质申请、账号使用是否合理

系统应执行的动作：

1. 从文档中识别业务动作和业务场景
2. 将文档表述映射到系统标准业务场景
3. 校验文档与 E-Flow 是否一致
4. 将业务场景作为后续权限合理性审视的输入

实施判断：

- `需求大类` 适合硬规则 + 枚举映射
- `业务场景` 更适合语义归一 + 场景一致性判断

---

### 3.2 网银平台基本信息

涉及字段：

- 平台编码
- 网银平台名称
- 平台入口
- 支持语言
- 银行名称
- 银行简称
- 银行所在国家/地区
- 分支银行名称

业务特征：

- 平台类字段通常不是文档中按同名方式直接出现
- 往往需要从表头、银行名称、中英文名称、分行、国家等组合推断

系统应执行的动作：

1. 从文档表头和正文中提取银行平台相关信息
2. 建立平台归一化规则
3. 将文档中银行/分行/国家表达映射为内部平台对象
4. 校验平台识别结果是否足以支持后续审查

实施判断：

- 这里不适合单纯精确比对
- 更适合作为“字段映射与归一层”的代表场景

建议：

- 平台识别应输出置信度
- 当平台无法可靠识别时，应触发人工确认，而不是继续默认为低风险

---

### 3.3 基础信息

涉及字段：

- 申请人
- 申请人部门
- 需求部门
- 权限类型
- 申请原因
- 其他支撑文档

其中当前最关键的是：

- `申请人`
- `权限类型`

#### 申请人

文档映射：

- 可能体现为“联系人”
- 也可能体现为“申请人”

检查要求重构：

1. 提取文档中的申请人/联系人候选值
2. 判断其是否可映射为系统中的申请人
3. 与 E-Flow 中申请人字段做一致性比对
4. 若多文档中申请人表述不一致，标记交叉冲突

实施判断：

- “是否存在该人名”适合硬规则
- “联系人是否等价于申请人”属于语义映射判断

#### 权限类型

文档映射：

- 当前映射到“表3-客户证书信息 操作权限”

检查要求重构：

- 不应单独孤立检查
- 应与“用户信息-权限范围”统一建模

---

### 3.4 用户信息

涉及字段：

- 名义用户
- 用户所属部门
- 权限子类
- 当前平台已有介质
- 权限范围
- 属性
- 保管信息
- 领用信息
- 权签人类别

这是整张表中最关键的模块之一。

#### 名义用户

文档映射：

- 表3-持有人姓名

检查逻辑重构：

1. 提取文档侧名义用户/持有人姓名
2. 比对系统用户姓名、工号、部门、证件等基准信息
3. 对所有申请材料执行多文档交叉校验
4. 执行完整性检查

推荐输出项：

- 一致
- 不一致
- 缺失
- 多文档冲突
- 需人工确认

#### 当前平台已有介质

这是当前业务逻辑里非常典型的“条件判断字段”。

Excel 已经定义了清晰的场景分支：

1. 文档有介质信息，系统也有，且一致
   - 通过
2. 文档有介质信息，系统也有，但不一致
   - 需进一步确认
3. 文档有介质信息，系统无值
   - 信息不一致，需人工确认
4. 文档无介质信息，系统有值
   - 存在介质遗漏风险，需人工确认
5. 文档无介质信息，系统也无值
   - 风险较低

这是后续非常适合写成代码判定矩阵的一类逻辑。

#### 权限范围

这是当前全表最核心的检查项。

Excel 中已明确规定：

- 系统统一权限范围只有四大类
  - 授权
  - 支付
  - 查询
  - 上载

检查要求应重构为标准流程：

1. 从文档中提取所有权限表述与勾选项
2. 将原始权限表述归一到四大类标准权限
3. 从 E-Flow 中读取标准权限范围
4. 按四大类逐项比对
5. 输出：
   - 一致项
   - 缺失项
   - 冗余项
   - 差异项
6. 执行合规合理性审视：
   - 是否超配
   - 是否与业务场景匹配
   - 是否存在冲突

实施判断：

- “四大权限是否命中”可以做成半结构化规则
- “原始表述如何归一到四大权限”属于语义映射
- “是否超配/合理”属于语义审视

---

### 3.5 介质信息

涉及字段：

- 名义用户
- 申请新介质类型
- 选择空白介质

#### 申请新介质类型

文档要求说明了：

- 需从申请文档和附件中提取用户申请的介质类型
- 还需关注该介质是否为实体介质

这说明当前系统的 `media` 模型还不够丰富，后续至少要考虑：

- 介质类型
- 介质编号
- 是否实体介质
- 是否新申请
- 是否加挂

#### 选择空白介质

这是另一类典型的条件矩阵检查项。

Excel 中已明确规则：

1. 系统字段有值
   - 检查文档是否明确表述使用已有空白介质
   - 若文档与系统一致，则通过
   - 若不一致，则标记异常
2. 系统字段为空
   - 检查文档是否提及空白介质
   - 若文档提及而系统为空，则信息不一致
3. 多文档之间还需做交叉比对

实施判断：

- 这类逻辑高度适合“硬规则判定 + 文档事实提取”的组合
- 不适合完全交给泛化 semantic check

---

### 3.6 权限信息

涉及字段：

- 名义用户
- 权限子类
- 账号
- 公司编码
- 账户中文名称
- 账户外文名称
- 账户状态
- 账户性质/子别
- 收付款类型
- 支付号
- 查询号
- 权限范围

#### 账号 / 账户名称

Excel 中已经定义出完整检查逻辑：

1. 提取文档中的银行账号、账户名称
2. 与系统电子流中的账号、账户名称做一致性比对
3. 对多份材料的账号信息做交叉核验
4. 执行完整性与合规审视
5. 关注“账号与业务主体不匹配”等异常

实施判断：

- 账号精确比对：硬规则
- 账户名称中英文对照、主体匹配：语义映射 + 规则
- 多文档交叉一致性：交叉校验器

#### 账户状态

当前表里的要求远比“字段相等判断”更复杂：

1. 正向比对
   - 文档中的账户状态 vs 系统备案状态
2. 反向审视
   - 系统状态为失效，但文档仍按生效状态办理
3. 多文档交叉比对
   - 多份材料中的状态描述是否统一

这类逻辑不能只靠简单 MISMATCH，需要显式保留“反向风险”标签。

## 5. 哪些逻辑适合硬规则，哪些适合 Prompt

### 5.1 适合硬规则/代码矩阵的逻辑

- 账号精确一致性
- 账户状态枚举一致性
- 介质有/无的条件组合判定
- 空白介质字段条件判定
- 字段缺失/空值判断
- 权限四分类逐项对比结果汇总
- 多文档间同一字段是否一致

### 5.2 适合 Prompt/语义归一的逻辑

- 联系人是否可映射为申请人
- 银行表述如何归一到平台对象
- 权限原文如何归一到四大权限分类
- 权限申请是否与业务场景合理匹配
- 账户名称与业务主体是否匹配
- 文档中是否存在非标备注、附加条款、隐含风险

### 5.3 需要“抽取 + 规则”组合的逻辑

- 当前平台已有介质
- 空白介质使用
- 账户状态反向风险
- 权限范围超配
- 多文档账号与申请人交叉校验

## 6. 当前系统与这张表的主要差距

### 6.1 当前系统更像“泛化抽取 + 泛化比对”

当前 `prompts.json` 中的 Prompt 更偏：

- 通用抽取
- 通用语义比对
- 通用总结

而这张业务表要求的是：

- 字段簇级抽取
- 字段簇级映射
- 字段簇级检查
- 条件矩阵判定
- 多文档交叉校验

### 6.2 当前系统缺少“映射归一层”

当前系统直接从文档抽取结果进入比对层，但这张表显示很多字段必须先做：

- 字段映射
- 语义归一
- 业务语义翻译

### 6.3 当前系统缺少“反向审视”与“人工确认触发”的显式机制

当前很多检查还只有：

- MATCH
- MISMATCH

但这张表要求的判定明显更丰富：

- 一致
- 不一致
- 风险较低
- 信息遗漏
- 需人工确认
- 存在反向异常

---

## 7. 面向后续 Prompt 重构的最小结论

在真正重写 Prompt 前，至少要先把系统任务拆成下面四类：

1. 文档事实抽取
2. 字段映射与语义归一
3. 字段级检查与判定
4. 多文档交叉校验与总报告汇总

否则，单靠继续加长现有 `semantic_risk_analyzer` 的 prompt，只会让任务继续混杂，效果不会稳定。

---

## 8. 多场景兼容补充结论

在纳入“权限注销”对照表后，可以进一步确认：

- 这套检查逻辑不能只服务“权限开通”
- 同一套字段簇和检查类型需要兼容：
  - 开通
  - 注销
  - 介质相关办理
  - 后续可能出现的其他变更类场景

因此，这份检查逻辑表后续使用时应遵循：

1. 字段簇统一
2. 检查类型统一
3. 场景策略分离

也就是说：

- `业务场景` 不是普通字段，而是整个检查策略的分流器
- `权限范围` 的判断不能只有“是否超配”，还要支持注销场景下的“是否少销、误销、对错对象”
- `介质信息` 的判断不能只看“有无”，还要看“本次动作是新增、保留、注销还是不处理”

简化后的系统原则是：

> 统一能力骨架，按场景加载不同判定策略。
