# 银行审计自动化系统：项目实施架构文档 (v1.0)

本项目旨在通过 OCR、大语言模型 (LLM) 和逻辑校验算法，全自动执行银行对公业务（如开户、修改、注销）的申请表审计，确保客户提交的纸质/电子表单与身份证明文件信息一致且符合银行规则。

---

## 1. 核心技术栈 (Technology Stack)
- **后端框架**: FastAPI (Python 3.12+)
- **OCR 引擎**: PaddleOCR (PaddleX 3.4.0) - 用于图像文字识别
- **PDF 解析**: PDFPlumber - 用于原生 PDF 文本直接提取
- **AI 引擎**: LLM (OpenAI SDK/Kimi/本地私有化接口) - 用于长文本的结构化提取
- **前端界面**: Vanilla JavaScript / HTML5 / CSS3 - 实现流畅的上传与交互体验

---

## 2. 系统核心流程 (Workflow)

### 2.1 数据提交与预处理
1. **文件上传**: 用户上传“银行申请表”（Word/PDF/图片）和“经办人/法人证件”（图片/PDF）。
2. **多模态解析**:
   - **图像/扫描件**: 调用 PaddleOCR 获取位置与文字。
   - **原生 PDF**: 优先调用 PDFPlumber 提取文本，效率与准确度更高。
   - **ID 证件**: 专用图像解析引擎，支持身份证、护照 (MRZ 码) 识别。

### 2.2 结构化提取 (Structured Extraction)
系统调用 LLM（如 Kimi），通过构造严谨的 `system_prompt` 和 `JSON Schema`，将非结构化的 OCR 文本转化为统一的 JSON 对象：
- **Company**: 企业名称、统一信用代码、法定代表人、行业信息等。
- **Operator**: 经办人姓名、证件类型/号码、角色、过期日期、联系电话。
- **Account**: 拟新开/修改的业务账号、所属分行。
- **Permissions**: 授权级别、单笔/每日交易限额。
- **Activity**: 申请业务的具体清单（功能勾选项）。

### 2.3 逻辑审计与规则校验 (Logic Audit)
解析完成后，逻辑引擎会执行“交叉审计”：
1. **证件比对**: ID 处理结果中的“姓名/证号”与“申请表”中对应字段是否一致。
2. **时效检查**: 证件是否已过期。
3. **关键缺失**: 是否缺少法人签名或必要的勾选项。
4. **限额预警**: 是否设置了超常规的交易限额。

## 4. 近期重大改进回顾

1. **PDF 原生支持**: 实现了对 PDF 文件（原生/扫描混合）的全面解析支持。
2. **解析鲁棒性提升**: `_parse_eflow` 逻辑重构，能够自适应各种不规范的 AI 返回（如列表降维、字段名混淆）。
3. **Schema 扩展**: 针对银行细化审计需求，新增了指派人 (Handler)、行业 (Industry)、多级权限与限额字段的高精度提取。
4. **单元化案例**: 建立了完整的 `test_data` 案例集，确保即使 LLM 提示词变更，核心逻辑依然稳定可控。

---

## 5. 后续演进方向
- **离线化部署**: 深度优化 PaddleOCR 初始化性能，支持在无外网环境下稳定运行。
- **更强的表格理解**: 针对超大、超复杂 Word 表格，引入专门的 Layout 文档还原引擎。
- **自动纠错建议**: 审计失败时，由 LLM 生成详细的客户引导建议（如：“请重新上传有效期内的证件”）。
