# 银行通道/权限预审系统设计白皮书 (V3)
**版本**: V3.0
**更新重点**: 解决多银行无模板化提取难题、重构多文档流式审核架构、引入“硬比对+软审视”双引擎。

## 1. 业务痛点与升级动机
在对接全球数百家银行时，由业务人员通过“坐标”或“黑名单正则”去配置固定模板来抽取内容，是不现实且高昂的。银行申请表、授权书常常排版迥异、语言繁乱。
此外，现实业务开户/权限申请中，**并不是“E-Flow”对比“单份 Word”** 这么简单的单线联系；往往涉及**多份申请件（表单、证照）的合并审核与交叉验证**。

## 2. 核心架构突破

### 2.1 抛弃模板，走向泛化 (Template-Free Generalization)
我们将复杂的 44 个银行字段归纳凝练为【实体】与【关系】：
- **平台信息 (PlatformInfo)**: 这张表是给哪家银行的、哪个系统的。
- **业务动作 (Business Activity)**: 开通、修改、还是注销。
- **目标载体 (MediaInfo)**: 这个权限将以 U盾、Token 还是密码形式交付。
- **权限边界 (PermissionScope)**: 【核心四要素】—— 一切银行业务均向此收拢：**授权(Authorize)、支付(Payment)、查询(Query)、上载(Upload)**。
> **AI 的角色转变**：在 V3 中，Prompt `global_document_extraction` 不再寻求文档特定的“表3”这种字眼，而是去通篇**感知和翻译**，将文中出现的任意权限描述投射到四大基本权限分类上。

### 2.2 流式多文档并行比对路由 (Multi-Doc Async Pipeline)
之前的架构在 `routers/audit.py` 中将 EFlow, Word, OCR 的结果一次性揉在一块比对。
**V3 将其升级为真正的工作流 (Workflow)**：
1. **统一锚点**：提取 `eflow.json` 解析出的《黄金基准》。
2. **切片审阅 (ForEach)**：无论客户传了 3 份 PDF 还是 5 份 JPG。系统对它们进行各自独立的 `Extraction (抽取) -> Hard Compare (硬比对) -> Semantic Compare (软审视)` 单线流。
3. **沉淀汇聚**：拿到 `[DocReport1, DocReport2...]` 的独立体检报告。
4. **十字路口交叉与总结**：`reporter.py` 将全部报告合并，检查“表A和表B的账号不同”等连带风险。并由 LLM 大脑输出唯一的 `Global Summary` (包含 `risk_insights`)。

### 2.3 “钢筋混凝土”架构：“硬比对”与“软推理”的结合
把“代码”和“AI”用在刀刃上。
- **Hard Comparator (代码级防虚伪)**：处理账号长度校验、统一社会信用代码对照、身份证白名单核验、人名的包含性比对。这是系统严肃合规性的底座（不可幻觉）。
- **Semantic Risk Analyzer (大模型风控防线)**：处理“权限范围是否超出电子流批复？”、“客户虽然账号写对了但是否擅自勾选了境外付款？”等强业务逻辑审查。

## 3. 系统模型沉淀 (Schema Overview)
本次最重要的财产重组都在 `schemas.py` 中，支持了细粒度的属性嵌套（比如同一个电子流下包含多个经办人、多套权限配比）。

- `EFlowData` & `DocExtractedData` 分离。
- `UserPermission` 成为核心细胞块，包裹了单体用户的四大权限及使用限额。
- 采用局部到全局的逐级汇报结构：`CheckResult -> DocAnalysisReport -> AuditReport`。

## 后记
这套体系允许我们在未来无缝平移到新银行，因为大模型作为“阅读理解引擎(MUC)”只需要认得四大权限，而非认识上百家银行的表单格子。
