# 网银权限预审：场景化规则矩阵

**适用范围**：当前基于“权限开通”和“权限注销”两类业务对照表整理。  
**文档目标**：把前面的业务理解、字段簇设计、检查逻辑和多场景兼容思路，进一步收敛成一份可直接指导后续重构与开发的规则矩阵。  
**核心原则**：

- 统一字段簇
- 统一检查类型
- 场景策略分离
- 硬规则与语义判断分层
- 明确保留“需人工确认”分支

---

## 1. 这份矩阵解决什么问题

前面的文档已经解决了“看懂业务”和“看懂规则来源”的问题。  
这份矩阵进一步解决的是：

> 当系统真正开始重构时，如何把“同一个字段簇”在不同业务场景下的检查逻辑写清楚。

它回答 5 个问题：

1. 这个字段簇在系统里为什么存在
2. 在 `OPEN` 和 `CANCEL` 场景下分别重点检查什么
3. 风险是往哪个方向偏移
4. 更适合放在硬规则、Prompt 还是组合层
5. 哪些情况必须触发人工确认

---

## 2. 统一场景定义

当前矩阵先覆盖两个已确认场景：

- `OPEN`
  - 包括权限开通、介质开通、权限加挂等“新增/扩展”导向的办理行为
- `CANCEL`
  - 包括权限注销、介质注销、取消已有权限对象等“撤销/解除”导向的办理行为

后续可自然扩展为：

- `MODIFY`
- `ATTACH`
- `REPLACE`

但当前文档先不展开。

---

## 3. 统一检查类型定义

全场景通用的检查类型保持不变：

- `consistency`
  - 一致性比对
- `completeness`
  - 完整性检查
- `reverse_review`
  - 反向异常审视
- `cross_doc`
  - 多文档交叉校验
- `compliance_review`
  - 合规合理性审视
- `manual_confirmation`
  - 需人工确认

---

## 4. 字段簇级场景规则矩阵

## 4.1 业务场景簇

覆盖字段：

- 需求大类
- 业务场景
- 办理目的
- 操作类型

### 业务意义

这是整条预审链路的分流器。  
后续所有规则都依赖它来判断“当前到底是在新增、加挂、注销、解除什么”。

### `OPEN` 场景

重点检查：

- 文档是否明确表达为开通/新增/加挂
- 办理目的是否支持本次新增权限或新增介质
- 文档与电子流的场景是否一致

风险方向：

- 文档写成注销、停用、解除
- 文档目的与新增申请不匹配
- 文档中存在模糊办理动作

### `CANCEL` 场景

重点检查：

- 文档是否明确表达为注销/取消/撤销/停用
- 办理目的是否真的指向“取消对象”
- 文档与电子流的注销对象和注销范围是否一致

风险方向：

- 表面写注销，但实际描述更像变更或开通
- 注销目的不清
- 文档场景与系统场景矛盾

### 推荐实现层

- 抽取：Prompt
- 场景归一：Prompt + 枚举映射
- 一致性校验：规则 + Prompt

### 人工确认触发

- 文档同时出现“开通”和“注销”语义
- 文档办理目的不清晰
- 多份附件对场景表述不一致

---

## 4.2 平台识别簇

覆盖字段：

- 平台编码
- 平台名称
- 银行名称
- 银行简称
- 国家/地区
- 分支银行
- 平台入口
- 支持语言

### 业务意义

平台对象决定后续材料解释上下文。  
同一套权限术语在不同银行平台上可能对应不同含义。

### `OPEN` 场景

重点检查：

- 文档是否能可靠识别出办理银行/平台
- 平台对象是否与系统电子流一致
- 是否存在跨银行、跨分行错配

### `CANCEL` 场景

重点检查：

- 被注销的对象是否属于正确平台
- 是否存在“注销 A 平台权限，但文档写成 B 平台”

### 风险方向

- 平台识别不稳定
- 分支银行信息错配
- 目标平台不明确

### 推荐实现层

- 文档识别：Prompt
- 归一化：Prompt
- 置信度控制：规则

### 人工确认触发

- 平台识别置信度低
- 银行名称、简称、分支行组合不自洽
- 多文档识别到不同平台对象

---

## 4.3 申请人与主体簇

覆盖字段：

- 申请人
- 联系人
- 申请人部门
- 需求部门
- 名义用户
- 用户所属部门
- 当前持有人

### 业务意义

这是“谁在申请、谁持有、谁是对象、谁对材料负责”的基础身份层。

### `OPEN` 场景

重点检查：

- 申请人/联系人是否可映射为系统申请人
- 名义用户是否与系统权限对象一致
- 用户与部门信息是否完整
- 多文档之间人员信息是否统一

风险方向：

- 联系人与申请人混淆
- 名义用户与权限对象不一致
- 多文档人名/工号/部门冲突

### `CANCEL` 场景

重点检查：

- 被注销权限/介质的对象是谁
- 名义用户与当前持有人是否对应
- 注销对象是否与系统登记用户一致
- 多文档是否对同一注销对象表述一致

风险方向：

- 注销对象写错人
- 当前持有人与名义用户不对应
- 申请人在办理 A 用户，文档却写成 B 用户

### 推荐实现层

- 人员抽取：Prompt
- 字符级一致性：硬规则
- 联系人/申请人映射：Prompt
- 多文档交叉：规则 + Prompt

### 人工确认触发

- 联系人和申请人都出现，但无法判定谁是系统申请人
- 当前持有人与名义用户存在部分重叠但不完全一致
- 多文档对同一对象的部门/工号表述不统一

---

## 4.4 权限定义簇

覆盖字段：

- 权限类型
- 权限子类
- 权限范围
- 权签人类别

### 业务意义

这是整个项目里最核心的业务对象簇之一。  
它决定“系统最终允许该对象做什么”。

### `OPEN` 场景

重点检查：

- 文档中申请的权限范围是否与系统批准一致
- 是否存在超配
- 是否存在冗余权限
- 是否与业务场景匹配
- 权限子类与权限范围是否自洽

主要风险：

- 超配
- 多申请
- 权限子类与范围不匹配
- 文档中的权限对象超出电子流批准范围

### `CANCEL` 场景

重点检查：

- 文档中声明要注销的权限是否与系统现有权限完全对应
- 注销范围是否过大
- 注销范围是否过小
- 是否注销错对象
- 权限子类、权限范围、注销对象是否一致

主要风险：

- 误销
- 少销
- 注销对象错配
- 注销范围与系统登记权限不一致

### 推荐实现层

- 原始权限表述抽取：Prompt
- 四大权限归一：Prompt
- 四大权限逐项比对：规则
- 是否超配/少销/误销：规则 + Prompt

### 人工确认触发

- 原始权限语句无法稳定归一到四大权限
- 文档既像全量注销，又像部分注销
- 权限子类与权限范围存在轻微冲突但无法直接判死

### 推荐输出标签

- `missing`
- `redundant`
- `excessive`
- `insufficient`
- `wrong_target`
- `conflict`

---

## 4.5 介质簇

覆盖字段：

- 当前平台已有介质
- 申请新介质类型
- 介质类型
- 介质编号
- 是否注销介质
- 介质状态
- 选择空白介质

### 业务意义

介质簇不是单纯字段，而是“权限对象的载体状态层”。  
它经常决定本次业务究竟是新增、加挂、注销还是保留。

### `OPEN` 场景

重点检查：

- 文档是否申请新介质
- 文档是否使用已有介质加挂权限
- 文档介质信息与系统已有介质是否一致
- 是否存在空白介质复用

主要风险：

- 文档有介质，系统无值
- 系统有介质，文档无值
- 文档介质与系统介质不一致
- 空白介质使用逻辑不清

### `CANCEL` 场景

重点检查：

- 系统是否存在介质
- 文档是否要求注销介质
- 若系统有介质但文档写“无需注销”，是否合理
- 文档中的介质类型、编号、注销动作是否与系统一致

主要风险：

- 销错介质
- 有介质但未注销
- 无介质却申请注销
- 介质与权限对象不对应

### 推荐实现层

- 介质事实抽取：Prompt
- 是否存在/条件矩阵判定：规则
- 动作判断：规则 + Prompt

### 人工确认触发

- 系统存在介质，文档无任何介质表述
- 文档写“无需注销介质”，但场景整体更像应注销
- 文档介质对象与系统对象部分匹配

### 推荐动作字段

- `media_exists`
- `media_type`
- `media_id`
- `action_on_media`
- `action_consistency`

## 4.6 账户簇

覆盖字段：

- 账号
- 账户中文名称
- 账户外文名称
- 账户状态
- 账户性质/子别
- 收付款类型
- 支付号
- 查询号

### 业务意义

账户簇承载本次权限或介质动作作用到的资金对象，是资金安全边界的一部分。

### `OPEN` 场景

重点检查：

- 文档账号是否与系统一致
- 账户名称是否与业务主体匹配
- 账户状态是否允许本次开通/加挂
- 多文档账户信息是否统一

主要风险：

- 账号填写错误
- 账户名称与主体不一致
- 失效账户仍被用于新增申请
- 多文档账号冲突

### `CANCEL` 场景

重点检查：

- 被注销权限对应的是哪个账户
- 账户状态是否支持本次注销办理
- 注销对象是否错指向其他账户
- 多文档是否对同一账户表述一致

主要风险：

- 注销错账户
- 系统失效账户与文档表述不一致
- 注销材料中的账户对象与电子流不对应

### 推荐实现层

- 账号精确比对：硬规则
- 账户名称主体匹配：Prompt + 规则
- 账户状态反向审视：规则
- 多文档交叉：规则

### 人工确认触发

- 中文/外文账户名称仅部分对应
- 账户状态在不同附件中冲突
- 账号一致但账户名称主体存在歧义

## 5. 场景化风险方向矩阵

为了防止系统总是默认“开通式思维”，这里单独列一张风险方向表。

| 字段簇 | `OPEN` 主要风险方向 | `CANCEL` 主要风险方向 |
|---|---|---|
| 业务场景簇 | 场景写错、申请目的不符、办理动作模糊 | 注销目的不清、注销动作错判、场景矛盾 |
| 平台识别簇 | 办错平台、跨银行错配 | 注销对象平台错配 |
| 申请人与主体簇 | 申请人/名义用户不一致 | 注销对象写错人 |
| 权限定义簇 | 超配、冗余、多申请 | 少销、误销、对错对象 |
| 介质簇 | 误申请新介质、加挂对象不清 | 有介质未注销、无介质误注销、动作不清 |
| 账户簇 | 开通到错账户、失效账户误用 | 注销到错账户、账户对象错配 |

## 6. 场景化实现建议矩阵

| 能力层 | 是否统一实现 | 是否需要场景策略 | 说明 |
|---|---|---|---|
| 文档接入与预处理 | 是 | 否 | 开通/注销共用 |
| 文档事实抽取 | 是 | 是 | 抽取骨架统一，但动作对象和重点字段受场景影响 |
| 字段映射与归一 | 是 | 是 | 权限、介质、业务动作在不同场景下归一方向不同 |
| 单文档硬规则检查 | 是 | 是 | 规则引擎统一，但判定矩阵按场景分支 |
| 多文档交叉校验 | 是 | 是 | 交叉对象相似，但冲突解释随场景变化 |
| 报告汇总与表达 | 是 | 是 | 报告骨架统一，但风险话术必须场景化 |

## 7. 对后续系统重构的最小落地建议

如果后面要顺着这个思路开发，我建议最小实施顺序如下。

### 第一步：数据结构补齐场景维度

至少补齐这些概念：

- `scenario_type`
- `action_type`
- `action_target`
- `action_scope`

### 第二步：检查输出补齐场景维度

每条检查结果建议都显式带：

- `field_group`
- `check_mode`
- `scenario_type`
- `manual_confirmation_required`
- `reason_code`

### 第三步：规则引擎改成“统一入口 + 场景分支”

例如：

```text
check_permission_scope(field_data, scenario_type)
check_media(field_data, scenario_type)
check_account(field_data, scenario_type)
```

而不是：

- `check_open_permission_scope`
- `check_cancel_permission_scope`

到处复制一套。

### 第四步：Prompt 也做参数化

Prompt 不要为每个场景复制一整套长文本，而应尽量做成：

- 统一 Prompt 骨架
- 注入 `scenario_type`
- 注入场景关注重点

## 8. 一句话版本的开发原则

后续开发时，最值得坚持的一句话原则是：

> 先把系统做成“可识别当前在处理什么场景、什么对象、什么动作”，再去谈字段抽取得有多完整、报告写得有多漂亮。

因为只要场景、对象、动作这三件事没建稳，开通和注销的逻辑就一定会互相串味。
